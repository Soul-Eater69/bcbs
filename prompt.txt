Yes — the logic is reusable. The problem is not the logic itself, it is the layering.

Right now the current code mixes too many responsibilities in one place:

your summary/index file both calls ingestion and builds/searches the FAISS store. get_meaningful_text_from_ticket_id() creates the Jira client, LLM, embedding client, config, calls ingest_ticket, writes a debug file, and returns observed.retrieval_text; then the same file also owns make_summary_document, build_langchain_faiss_summary_index, and search_langchain_faiss_summary_index.
the pipeline file mixes adapter construction, config building, record projection, and batch runtime/orchestration in one module. The OpenAI-compatible wrappers live beside config helpers, record builders, and the async batch runner.
one of the uploaded pipeline variants still has unresolved Git merge markers, so that version should not be used as the base for refactoring.

So yes, you can fix this cleanly.

What to keep vs what to move

Keep these as reusable domain/pipeline functions:

build_chunk_record
build_valuestream_record
build_config idea
the call to ingest_ticket
the summary FAISS helpers

Those are useful. The issue is that they are not isolated. process_ticket() currently does ingestion, projection, logging, and file persistence all together.

Best refactor shape

Split into 4 layers.

1) services/ingestion_service.py

This should be the only place that knows how to call ingest_ticket.

It should do one job:

doc = await ingest_one_ticket(ticket_id, deps, cfg)

Return the raw ingestion result, nothing else.

2) projections/jira_records.py

Move pure transformation logic here:

build_chunk_record
build_valuestream_record
extract_retrieval_text(doc)
later: build_ticket_summary_input(doc)

These are pure functions from doc -> shape.

3) factories/runtime.py

Move environment/runtime creation here:

build_config()
create_memory_indexes()
build_llm_client()
build_embedding_client()

Right now those helpers are mixed into the batch runner.

4) jobs/batch_ingest.py

This becomes thin orchestration only:

loop tickets
call service
project outputs
persist files
log

That means process_ticket() gets much smaller.

What is tightly coupled today

This is the exact coupling problem:

In your summary file

get_meaningful_text_from_ticket_id() hardcodes:

how to create Jira client
which model to use
which embedding client to use
which config flags to use
that a debug file should be written
that only observed.retrieval_text should be returned

So you cannot easily reuse it for:

batch mode
tests
alternate config
summary generation
local-only runs without rewriting the function
In your pipeline file

process_ticket() hardcodes:

local file paths
persistence names
projection logic
logging behavior
ingestion call
batch/runtime dependencies passed separately

So it is reusable only if you want that exact workflow.

What I would change first
Step 1: introduce one dependency container
from dataclasses import dataclass
from typing import Any

@dataclass
class IngestionDeps:
    jira_client: Any
    llm_client: Any | None
    embedding_client: Any | None
    coarse_index: Any
    fine_index: Any
    metadata_index: Any
    supervision_store: Any

Now ingest_ticket dependencies travel together.

Step 2: make one raw ingestion function
async def ingest_one_ticket(
    ticket_id: str,
    deps: IngestionDeps,
    cfg: JiraIngestionConfig,
    storage_dir: str | None = None,
) -> dict:
    return await ingest_ticket(
        ticket_key=ticket_id,
        jira_client=deps.jira_client,
        coarse_index=deps.coarse_index,
        fine_index=deps.fine_index,
        metadata_index=deps.metadata_index,
        supervision_store=deps.supervision_store,
        config=cfg,
        llm_client=deps.llm_client,
        embedding_client=deps.embedding_client,
        storage_dir=storage_dir,
    )

Now the pipeline call is decoupled from file writing and downstream projection.

Step 3: make pure projection functions
def project_ticket_outputs(ticket_id: str, result: dict) -> tuple[list[dict], dict]:
    obs = result.get("observed", {})
    meta = obs.get("metadata", {})
    chunks = [build_chunk_record(c, ticket_id, obs, meta) for c in obs.get("chunks", [])]
    vs_map = build_valuestream_record(ticket_id, result)
    return chunks, vs_map


def extract_retrieval_text(result: dict) -> str:
    return str(result.get("observed", {}).get("retrieval_text") or "")

Now your summary-index flow can reuse the same ingestion result without re-running logic.

Step 4: make persistence separate
def persist_ticket_outputs(ticket_dir: Path, ticket_id: str, chunks: list[dict], vs_map: dict) -> None:
    dump_json(ticket_dir / "07_chunks.json", {
        "ticket_id": ticket_id,
        "mapped_value_stream_ids": vs_map.get("valueStreamIds", []),
        "mapped_value_stream_names": vs_map.get("valueStreamNames", []),
        "chunk_count": len(chunks),
        "chunks": chunks,
    })
    dump_json(ticket_dir / "08_valuestream_map.json", vs_map)

Now batch runtime can call persistence, but summary generation can skip it.

The important design rule

Do not let summary generation call ingest_ticket directly.

Right now your summary file does exactly that.

Instead do this:

ingestion service returns raw result
projection layer derives:
retrieval text
chunk records
value stream map
summary input
summary index layer only consumes projected summary docs

That is the decoupling.
