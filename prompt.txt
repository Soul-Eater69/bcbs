# Architecture: Summary-First Retrieval for Idea Cards

## Goal

Improve value-stream inference for a new Jira idea card / PPT by moving away from noisy raw historical chunk retrieval and toward a **summary-first retrieval architecture**.

The main problem with the earlier approach is that semantic search over raw historical chunks tends to match broad enterprise-healthcare language across many tickets. That leads to:

- too many historical ticket hits,
- many semantically similar but weakly relevant chunks,
- large LLM context windows,
- poor evidence density,
- unstable value-stream selection.

This design introduces:

1. **New-card semantic summary generation**
2. **Historical ticket semantic summaries**
3. **A local LangChain + FAISS summary index**
4. **Late-stage raw chunk lookup only for shortlisted tickets**
5. **A cleaner separation between ingestion, projection, indexing, and orchestration**

---

## High-Level Architecture

```text
New Jira Idea Card / PPT
        |
        v
clean + normalize PPT text
        |
        v
generate semantic summary for new card
        |
        v
build retrieval text from semantic summary
        |
        v
search historical ticket summary FAISS index
        |
        v
retrieve top analogous historical tickets
        |
        +------------------------------+
        |                              |
        v                              v
retrieve KG / value-stream candidates  optional raw chunk lookup for top tickets only
        |                              |
        +--------------+---------------+
                       |
                       v
compact evidence package for LLM
                       |
                       v
final value-stream selection
```

---

## Why This Approach

### Problem with raw chunk retrieval

Earlier retrieval was based on historical raw ticket chunks. That causes weak discrimination because many tickets share generic language such as:

- member
- provider
- product
- engagement
- workflow
- digital
- solution
- growth

The vector retriever therefore tends to pull:

- many tickets,
- many chunks from the same broad domain,
- weakly differentiated evidence.

### Why summaries help

A semantic summary compresses raw ticket/PPT content into a clearer business representation:

- what the initiative is,
- which actors are involved,
- what functions are directly affected,
- what downstream processes are implied,
- what the business goal is.

This produces better retrieval because the embedding reflects **intent and business semantics**, not just shared jargon.

---

## Core Design Principles

### 1. Summaries for retrieval
Use semantic summaries as the primary representation for retrieving analog historical tickets.

### 2. Raw chunks for verification
Keep raw chunks, but only use them later:

- for evidence display,
- for traceability,
- for verifying borderline candidates,
- for final compact grounding.

### 3. Small, dense prompts
The LLM should not receive 15 tickets and 32 chunks. It should receive:

- the new card semantic summary,
- shortlisted value-stream candidates,
- top historical analog summaries,
- a few carefully selected raw snippets only if needed.

### 4. Decouple pipeline layers
Ingestion, transformation, indexing, and batch orchestration should be separate.

This summary should capture:

- short summary

## Step 2: Build Retrieval Text

The retriever often expects a single string query, not structured JSON.

So we convert the summary into a packed retrieval string.

### Example retrieval text

```text
title: Employer/member advocacy launch
Launch of a new employer/member health advocacy offering focused on navigation, engagement, and care coordination.
business goal: launch a new healthcare offering
actors: member, provider, employer
direct functions: product offering, engagement, care coordination
implied functions: provider setup, pricing, billing, IT enablement
change types: new product launch, operational rollout
domain tags: health advocacy, navigation, member experience
evidence: launch a new employer/member health advocacy solution | improve navigation and care coordination
```

This is the text embedded and used for similarity search.

## Step 4: Build Historical Summary FAISS Index

### Storage choice
For local experimentation, use:

- **LangChain FAISS**
- **1536-dimensional embeddings**
- local persistence on disk

### Why FAISS here

- fast local semantic search
- easy experimentation
- avoids Azure indexing while architecture is still evolving
- simple integration with LangChain retrievers later

### Vector representation

Each historical summary document is converted into retrieval text and embedded into a 1536-dimensional vector.

### Index contents

Each FAISS record corresponds to one historical ticket summary.

The document content stored in the vector store is the `retrieval_text`.

Metadata contains the structured fields for later inspection and downstream use.

## Query Path with the New Index

### Query flow

1. Clean the new PPT / card text
2. Generate new-card semantic summary
3. Build retrieval text from summary
4. Query local FAISS summary index
5. Retrieve top `k` historical tickets
6. Use these as analog evidence
7. Only then fetch raw chunks for a few top tickets if necessary

### Result shape

```json
[
  {
    "rank": 1,
    "score": 0.82,
    "ticket_id": "IDMT-19761",
    "title": "Employer/member advocacy launch",
```

## How Raw Chunks Still Fit

Raw chunks are still useful, but they move to a later stage.

### Raw chunks should be used for:

- evidence verification
- UI citation / traceability
- borderline candidate validation
- final support snippet extraction

### Raw chunks should not be used for:

- first-pass analog retrieval
- large prompt stuffing
- broad history-first reasoning

### Final pattern

- summary index = primary analog retrieval
- raw chunk data = optional evidence lookup for shortlisted tickets only

---

## Value-Stream Inference with This Design

The final value-stream selection should not depend on historical chunk retrieval alone.

Instead, combine:

1. **new-card semantic summary**
2. **candidate value-stream definitions / KG retrieval**
3. **historical analog summaries from FAISS**
4. **optional small raw evidence snippets**

This gives the LLM a compact and higher-quality evidence package.

### Final LLM input should look like:

- new card semantic summary
- shortlisted value-stream candidates
- top historical analog ticket summaries
- a few raw evidence lines only when needed

Not:

- full PPT text
- 15 historical tickets
- 32 raw chunks

## Recommended Pipeline Refactor

The current implementation has useful logic, but it is tightly coupled.

### Current coupling issues

The current summary/index file does all of the following in one place:

- creates Jira client
- creates LLM client
- creates embedding client
- builds config
- calls ingestion
- writes debug JSON
- returns retrieval text
- builds and queries FAISS

The pipeline file also mixes:

- runtime setup
- config construction
- record transformation
- output persistence
- async batch orchestration

This makes reuse hard.

---

## Proposed Layered Refactor

## 1. `runtime/` layer
Responsible for environment and client creation.

Examples:

- `build_config()`
- `build_llm_client()`
- `build_embedding_client()`
- `create_memory_indexes()`

## 2. `services/ingestion_service.py`
Responsible only for calling ingestion.

Example responsibility:

```python
result = await ingest_one_ticket(ticket_id, deps, cfg)
```

## 3. `projections/` layer
Pure transformation logic.

Examples:

- `build_chunk_record(result)`
- `build_valuestream_record(result)`
- `extract_retrieval_text(result)`
- `build_ticket_summary_input(result)`

## 4. `indexes/summary_faiss.py`
Responsible only for:

- converting summary docs to LangChain Documents
- building the FAISS store
- saving/loading the FAISS store
- searching the FAISS store

## 5. `jobs/batch_ingest.py`
Thin orchestration only.

Responsibilities:

- iterate tickets
- call ingestion service
- project outputs
- persist outputs
- log progress

## Suggested Implementation Sequence

### Phase 1: New card summary generation
- generate semantic summary for new PPT / idea card
- build retrieval text
- test summary quality manually

### Phase 2: Historical summary generation
- create summary docs for historical tickets
- persist summary docs locally

### Phase 3: Local FAISS index
- build LangChain FAISS index using summary docs
- query it using the new-card summary retrieval text
- inspect top hits

### Phase 4: Raw evidence integration
- for top 3–5 historical ticket hits only,
- fetch 1–2 best raw chunks if needed

### Phase 5: Final LLM evidence package
- combine:
  - new-card summary,
  - value-stream candidates,
  - top historical analog summaries,
  - few raw snippets

---

## What Success Looks Like

Compared to the previous raw-chunk retrieval approach, the improved system should show:

- fewer historical ticket matches,
- more believable top analog tickets,
- smaller final LLM prompts,
- higher evidence density,
- better distinction between direct and indirect value-stream support.

---

## Final Recommendation

### Keep
- ingestion logic
- chunk record builders
- value-stream mapping logic
- raw chunk persistence

### Add
- new-card semantic summary generation
- historical ticket summary generation
- LangChain FAISS summary index
- summary projection layer

### Change
- stop using raw historical chunks as the primary analog retrieval surface
- use raw chunks only after shortlisting historical tickets
- decouple ingestion from indexing and orchestration

---

## One-Line Mental Model

**Summaries retrieve. Raw chunks verify. The LLM adjudicates on compact evidence, not on a flood of historical text.**
